23 # The WatGUI Toolkit: Overview # {#intro}
24 
25 The WatGUI toolkit is a very basic X11-based graphical user interface
26 (GUI) toolkit. The intent of the toolkit is to give you an opportunity
27 to learn about the internals of a GUI toolkit through a relatively
28 small code base.
29 
30 The toolkit also demonstrates how to integrate a scripting engine with
31 an application. We have tightly integrated TinyScheme, a very small
32 Scheme interpreter with the toolkit. Briefly, the Scheme interpreter
33 allows you to directly interact with the running application, creating
34 new components on-the-fly, or adjusting existing components'
35 attributes. More documentation about the Scheme integration can be
36 found below.

68 ## Logging in Code ## {#logging_in_code}
69 
70 We're using the library's verbose logging features to help manage

72 are defined:
73 
74 + `LOG_TODO`
75 + `LOG_PAINT`
76 + `LOG_DEBUG`
77 + `LOG_INFO`
78 + `LOG_WARN`
79 + `LOG_ERROR`
80 + `LOG_FATAL`
81 
82 To log something, use normal C++ streams operators. For example:
83 
84  LOG_DEBUG << "Value returned: " << value;
85 
86 Note that you don't need an `endl` at the end of the logging
87 comment.

89 ## Logging Levels ## {#logging_levels}
90 
91 Google's logging library has two basic forms of logging macros, the
92 "normal" logging macro (`LOG(level)`) and the verbose logging macro
93 (`VLOG(level)`). The verbose logging macro provides greater
94 flexibility in terms of controlling what log messages are output at
95 runtime, so we use it for the TODO, paint, debug, info, and warning
96 levels. More information about how to control what is output is
97 provided in a section below.
98 
99 `LOG_TODO` logs sections of the toolkit that need to be
100 implemented by you. `LOG_PAINT` logs painting operations to debug
101 drawing to the screen. `LOG_ERROR` and `LOG_FATAL` are used to log
102 unexpected states of the running application.
103 
104 **When logging messages for your own benefit (i.e., during development
105 and during debugging), you should only use macros that use the VLOG
106 macros** (such as `LOG_PAINT`, `LOG_TODO`, `LOG_DEBUG`, `LOG_INFO`,
107 and `LOG_WARN`, or ones you define using VLOG). Using these macros
108 allows you to keep logging comments in your code when submitting it
109 because we can execute your binary with these logging messages
110 disabled (as described below).
111 
112 Error and fatal levels can also be inserted in your code, but should
113 only be used for true error/fatal conditions. These *will* be output
114 during marking. *Furthermore, LOG_FATAL messages will forcibly
115 terminate the application with a core dump, so use with care.*
116 
117 You are free to define new levels of logging for your own purposes --
118 use the debug, info, and warn macros as templates. Any new logging
119 levels you define using the VLOG macro will not be output during marking.
120 
121 ## Enabling Logging at Runtime and Specifying Which Files to Log ## {#logging_config}
122 
123 To output log messages, you need to set an environment flag
124 corresponding to the minimum logging level you wish to use. main.cpp
125 specifies that all logging messages should be output to stderr.
126 
127 To output all log messages at the TODO level or lower (where "TODO"
128 marks all the "to-do" points in the code), start the app from the
129 command line like this:
130 
131  GLOG_v=70 ./watgui
132 
133 `LOG_TODO` is mapped to verbose logging level 70, so by setting the
134 Google logging flag "v" to 70, it ensures all to-do messages (and all
135 other log messages with lower logging levels, including paint and debug
136 messages) are logged.
137 
138 *For your convenience, we have defined the shell script
139  `debug_app.sh`, which makes your project, then executes the app with
140  the most verbose logging level.* Use it like this:
141 
142  ./debug_app.sh a1 [any other command-line arguments to pass to the app]
143 
144 You can also enable logging on a per-file basis, which is *very* useful
145 when targeting a specific bug. To do so, use a comma-separated list of
146 file name globs, along with the logging level.
147 
148 For example, to show debug messages from all Event* files and
149 info messages from Component.cpp, set the environment variables like
150 this:
151 
152  GLOG_vmodule='Event*=50,Component=40' ./watgui
153 
154 Note the use of quotes (') to ensure the glob operator (*) is not
155 interpreted by the shell. *Also note that the comma-separated list of
156 files cannot have a space between items.*

176 # Unit Tests # {#unit_tests}
177 
178 We have included a number of unit tests to help you test/verify your
179 code, once you have made the necessary additions to the provided
180 source code. We are using [Google's unit testing framework](http://code.google.com/p/googletest/wiki/V1_7_Documentation) for testing.
181 
182 Unit tests will be automatically built by the Makefile and output as a
183 separate program called `x_unittests` where "x" will correspond to the
184 app name (e.g., `a1_unittests`). Executing this will run the unit
185 tests.
186 
187 Included in the build is a file MarkingUnitTests.cpp. This file is
188 empty and should remain so -- it will be replaced by our own unit
189 tests for marking purposes.
190 
191 Writing a unit test is pretty straightforward. In a C++ file, create a
192 new test using a TEST block:
193 
194  TEST(test_case_name, test_name) {
195  // test code
196  }
197 
198 For example, a very simple test of the EventQueue object:
199 
200  TEST(EventQueueTests, EventQueuing) {
201  EventQueue eventQueue;
202  EXPECT_EQ(0, eventQueue.GetNumEventsInQueue());
203  MemoryTrackingEvent* event = new MemoryTrackingEvent();
204  eventQueue.AddEventToQueue(event);
205  EXPECT_EQ(1, eventQueue.GetNumEventsInQueue());
206  delete event;
207  }
208 
209 See the
210  [Google test documentation](http://code.google.com/p/googletest/wiki/V1_7_Documentation)
211  or the included unit tests for more examples of how to write your
212  own unit tests.

226 # Scheme Interpreter # {#scheme}
227 
228 The interpreter is a very slightly modified version of TinyScheme,
229 which implements a subset of R5RS Scheme. A list of the Scheme
230 commands built-in to the interpreter can be found in
231 scheme/scheme_cmds.txt.
232 
233 The goal of having the interpreter is to help you experiment with the
234 GUI toolkit without needing to always write C/C++ code (thus avoiding
235 compile and test cycles). Once you have implemented the missing parts
236 of the toolkit, you should be able to perform many of the same GUI
237 operations you would do in C++ using the Scheme interpreter.
238 
239 *You are not required to understand any of the code provided that is
240 related to embedding the interpreter in the application. Nearly any
241 file that starts with "Scheme" can be ignored by you.*
242 
243 That file `cs349_scheme_tests.scm` serves to document many of the
244 functions available via the interpreter. `cs349_init.scm` provides
245 more functions of interest and relevance to your work.
246 
247 To execute any typed input, hit RETURN twice or press Ctrl-G. To quit
248 the interpreter (and the app), press Ctrl-D.
249 
250 **Note**: If you application terminates in an unexpected manner and
251  doesn't cleanly exit, the terminal can get a little unpredictable
252  since the readline library didn't properly shut down. The effects of
253  this are visible if you try to ssh into a remote host -- after it
254  asks for your password, it won't display anything else on the
255  terminal. Note that this is not the only symptom of this problem --
256  if things start getting a little weird and regular apps don't print
257  out on the console as expected, you may need to start a new
258  terminal.
259 
260 The Scheme interpreter will try to run any files passed to the
261 executable on the command line. For example, the following will
262 execute the Scheme demo/test file:
263 
264  ./a1 cs349_scheme_tests.scm
265 
266 To help debug Scheme scripts, you can turn tracing on/off while in the interpreter:
267 
268  (tracing 1) ;; Turns tracing on
269  (tracing 0) ;; Turns tracing off
